
 Modeling Report
1. Introduction: Understanding the Problem
The goal of this project was to build a machine learning model capable of predicting road accident injury severity in France. The target variable, injury_severity_label, contains four classes, making this a multiclass classification problem.
The task aligns with real-world applications such as risk assessment, public safety analytics, and emergency response prioritization, where accurate classification of severity can directly influence resource allocation and life-saving decisions.
Because the dataset is large (over 1.4 million rows) and the classes are imbalanced, the modeling journey required careful preprocessing, thoughtful model selection, and robust evaluation metrics. From the beginning, the focus was on macro F1 and balanced accuracy, since these metrics treat all classes fairly and reveal whether the model is learning minority patterns Ś especially important for severe accident categories.

2. Data Loading and Initial Exploration
The modeling process began with loading the master dataset:
    Ģ 1,412,032 rows
    Ģ 27 features
    Ģ A mix of categorical, numerical, and binary variables

    Ģ Rich categorical diversity (e.g., collision_label with 7 categories, manoeuvre_label with 9)

    Ģ A clean target variable with 4 severity classes
This exploration confirmed that the dataset was well-structured but required careful preprocessing to handle missing values, categorical encodings, and scaling.

    Ģ Train: 1,129,625 rows

This large training set provided a strong foundation for model learning, while the test set remained untouched for unbiased evaluation.

Before committing to any specific algorithm, the next step was to explore a wide range of models using LazyPredict Ś a tool that quickly benchmarks dozens of classifiers with minimal code.
However, because the dataset is extremely large (over 1.1M rows), LazyPredict initially froze on heavy models like:
    Ģ SVC
    Ģ GaussianProcessClassifier
    Ģ QuadraticDiscriminantAnalysis
    Ģ MLPClassifier

To avoid this, these models were excluded, and LazyPredict was run with a filtered list of classifiers.
Why LazyPredict was run on the imbalanced dataset
LazyPredict is meant for quick model comparison, not final training.
Running it on SMOTE-balanced data would distort results for simpler models.
Using the original imbalanced data gives a fair baseline across all algorithms.

8. Modeling Optimization with Feature Selection, PCA, and Controlled Oversampling
This section documents the complete optimization pipeline applied to the multiclass severity prediction model. The goal was to improve model efficiency, reduce dimensionality, address class imbalance, and enhance the modelÆs ability to detect minority classes (especially Killed and Hospitalized).
The optimization pipeline consisted of four major steps:
    1. Feature Selection using LightGBM Importance (85 ? 60)
    2. Dimensionality Reduction using PCA (60 ? 30)
    3. Oversampling using SMOTE and Controlled BorderlineSMOTE
    4. Training and evaluating optimized LightGBM models
8.1 Feature Selection using LightGBM Importance (85 ? 60)
A baseline LightGBM model was trained on the full 85-feature dataset. Feature importances were extracted and ranked.
Features with importance ? 100 were retained, resulting in a reduced set of 60 features.
Key Observations
    Ģ age and lane_width were the strongest predictors.
    Ģ Collision type, manoeuvre type, vehicle group, and road category dominated the top 20.
    Ģ Environmental and temporal features (night, weekend, weather) also contributed meaningfully.
This selection preserved the most informative predictors while removing low-impact noise.
8.2 PCA Dimensionality Reduction (60 ? 30)
To reduce redundancy and improve computational efficiency, PCA was applied to the scaled 60-feature dataset.
    Ģ Number of components retained: 30
    Ģ Explained variance: 77.95%
This dimensionality reduction preserved most of the predictive signal while enabling faster oversampling and model training.
8.3 Oversampling in PCA Space
Two oversampling strategies were applied to the PCA-transformed training data.
8.3.1 Standard SMOTE (Full Balancing)
SMOTE was applied to fully balance all four classes.
Class	Original Count	After SMOTE
0 ¢ Hospitalized	215,007	522,866
1 ¢ Slight Injury	363,089	522,866
2 ¢ Killed	28,663	522,866
3 ¢ Uninjured	522,866	522,866
    Ģ Training size after SMOTE: 2,091,464 samples
    Ģ PCA shape: (2,091,464 ū 30)
This approach aggressively balances all classes but risks introducing synthetic noise.
8.3.2 Controlled BorderlineSMOTE (Targeted Boosting)
To avoid excessive oversampling, a controlled strategy was applied:
Class	Original	Target
0 ¢ Hospitalized	215,007	350,000
1 ¢ Slight Injury	363,089	363,089
2 ¢ Killed	28,663	300,000
3 ¢ Uninjured	522,866	522,866
    Ģ Training size after BorderlineSMOTE: 1,535,955 samples
    Ģ PCA shape: (1,535,955 ū 30)
This method boosts minority classes while preserving the natural dominance of class 3.
8.4 Inverse PCA Transformation (30 ? 60)
To restore interpretability and compatibility with the original feature space, the oversampled PCA datasets were inverse-transformed back to 60 features.
Dataset	Shape	Avg Feature Variance
SMOTE (inverse)	(2,091,464 ū 60)	0.794
BorderlineSMOTE (inverse)	(1,535,955 ū 60)	0.791
Original scaled data	(1,129,625 ū 60)	1.000
Variance remained close to the original, confirming that PCA compression and reconstruction were effective.

Two optimized models were trained:

    2. LightGBM + Controlled BorderlineSMOTE (PCA 60 features)
